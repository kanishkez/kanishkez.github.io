Quantization

Before looking at quantization itself, we need to understand how numbers are actually stored in computer memory and GPU VRAM. It helps to look at how bits and bytes represent values, especially when dealing with floating point representations. Once we get a handle on that, we can see why we need to compress these numbers and how quantization makes LLMs run way faster without blowing up the memory.

Bits and Bytes of a Number

A bit is the smallest unit of a number, and it can be either 0 or 1.

A byte is made up of 8 bits and can store 256 different values, typically from 0 to 255 (unsigned) or -128 to 127 (signed).

Computers store negative numbers using two's complement.

Two's Complement

A 2's complement of a binary number is obtained by finding the 1's complement and then adding 1 to the result. The most significant bit represents the sign: 0 indicates positive and 1 indicates negative.

Decimal Binary Representation

To represent fractions like 1/2, 1/4, 1/8, etc., we use the power of 2 rule, just like any binary representation.

Normalized Scientific Binary

Normalized binary looks like this: $$1.xxxxx \times 2^{\text{exponent}}$$

So, if you need to move the binary point to the left, you add that many steps to the exponent of 2. If you need to move it right to get it into that form, you subtract that many steps from the exponent of 2.

Storing Decimals Using the Floating Point System

To store decimals, we use a system called floating point.

It consists of:

  • Sign bit: A single bit that tells you if the number is positive (0) or negative (1).
  • Exponent: A set of bits that determine the number's magnitude (how big the number is). The exponent is not stored directly as a signed number; instead, we use something called an exponent bias. This bias makes numerical comparison easier. For FP32, the bias is 127. We add the bias to the actual exponent to get the stored exponent: $$E_{\text{stored}} = E_{\text{actual}} + \text{bias}$$ So if the actual exponent is 3, we add 127 and store 130 instead.
  • Significand: It represents the precision bits. In normalized form, it is always the fraction bits (the numbers after the binary point).

Representing $13.25$ in FP32:

$$13.25 = 13 + 0.25$$ $$13 = 1101_2$$ $$0.25 = 0.01_2$$

So: $$1101.01_2 = 1.10101_2 \times 2^3$$

Rewriting in scientific binary:

  • Exponent = 3
  • Significand = $1.10101_2$

We calculate the stored exponent: $$\text{Stored Exponent} = 127 + 3 = 130$$ $$130 = 10000010_2$$

We ignore the leading 1 (since it is implicit in normalized form) and store only the fractional part in the significand: $$1.10101_2 \rightarrow \text{store } 10101000000000000000000_2 \quad (23 \text{ bits})$$

The sign bit is 0 since it is a positive number.

So, the final representation is:

Final bit pattern: 0 10000010 10101000000000000000000

The Tradeoff

The tradeoff between the bits assigned to the exponent or the significand is what varies across different floating point formats.

Floating Point Types

16 bit numbers (FP16 and BF16)

They both use 2 bytes (16 bits total) but prioritize different things:

  • FP16: Known as half precision. The significand (or mantissa) stores the detail of the number, while the exponent controls its scale. FP16 has a limited exponent (only 5 bits), so it cannot represent very large or very small values well. During LLM training, activations and gradients can grow or shrink rapidly. FP16 cannot represent these ranges very well, leading to overflow or underflow.
  • BF16: Developed to solve the range issue, it keeps the same exponent size as the FP32 standard (8 bits) but reduces the number of significand bits. It is the preferred standard for LLM training today.

8 bit numbers (FP8)

It comes in two formats: E4M3 (4 bit exponent, 3 bit significand) and E5M2 (5 bit exponent, 2 bit significand). It maintains the floating point exponent structure while compressing the float. This superior range of the exponent makes it a preferred format for quantizing activations (the outputs of network layers) since it handles extreme outliers better than integer formats like INT8.

32 bit numbers (FP32)

The standard "full precision" format for many calculations, offering a large range and high precision. This uses 4 bytes (32 bits) per number. The more bits we use to represent a number, the more precise it becomes.

Most LLMs use FP16 or BF16. Each parameter therefore takes 2 bytes (16 bits total).

For a 7 billion parameter model, if each parameter used 4 bytes in FP32, the total memory for weights alone would be 28 GB. In FP16 or BF16, that would be 14 GB. This footprint is still too large for many consumer GPUs, which is why quantization is used to shrink it further.

While the memory footprint is the initial barrier (making the model too big to fit), the ultimate purpose of quantization is to dramatically improve inference speed and throughput (how many requests a model can handle per second). Depending on the format and bits used, matrix multiplication can also run much faster, improving both training and inference speeds.

What is Quantization

The main aim of quantization is to reduce high precision weights into lower precision formats to improve speed and make large language models more practical, which is often better than just reducing the number of parameters in a model.

Quantization is the process of efficiently replacing high precision numbers (weights, biases, and activations) with lower precision ones with the goal of using less memory and less compute while keeping the same model behavior.

We map floating point values into discrete values represented by fewer bits, and then use a scale factor to convert back and forth between the smaller representation and an approximate floating point value.

How it Works

The core idea is that we convert real values to an integer range. We determine a minimum and maximum range for our weights, and all values are mapped into that range. This range can be chosen before quantization (static) or can change dynamically during runtime.

General formula: $$q = \text{round}\left(\frac{x}{\text{scale}}\right) + \text{zero point}$$ where:

  • $\text{round}()$ rounds to the nearest integer.
  • $\text{scale}$ scales the real value into the target integer range.
  • $\text{zero point}$ is an integer that represents the real value $0.0$. Since we are shifting the values into a specific unsigned or signed integer range, true zero shifts, and the zero point acts as that offset.

Scale Formula: $$\text{scale} = \frac{x_{\max} - x_{\min}}{q_{\max} - q_{\min}}$$ where:

  • $q_{\min}$ and $q_{\max}$ are the minimum and maximum values of the target integer range.
  • $x_{\min}$ and $x_{\max}$ are the minimum and maximum values of the floating point range being converted.

This scale helps us map an input into the target integer range along with the zero point offset.

Types of Quantization

Static and Dynamic Quantization

Static Quantization

Static quantization runs a calibration step through the model using a representative calibration dataset to measure the minimum/maximum values or the distribution of weights and activations. Using this data, we compute the scale and zero point, which are then frozen and used during inference. This requires an extra calibration step but usually preserves accuracy better.

Dynamic Quantization

We do not calculate the range ahead of time. Instead, we calculate the range on the fly for each batch, head, or layer during runtime, using the minimum and maximum values of the active batch. You dynamically recompute the scale and zero point for each input batch. This is simpler to set up since it does not require calibration data, but it adds runtime compute overhead.

Symmetric and Asymmetric Quantization

Symmetric Quantization

The quantization range is centered around zero, meaning we set the zero point to 0. This is commonly used for weights because weight distributions are often roughly symmetric around zero. The formula is simpler since there is no zero point offset.

Asymmetric Quantization

The quantization range is not centered around zero and can include a non zero minimum. The zero point shifts during mapping to align the real range with the integer range. This is commonly used for activations because activations (like ReLU or GeLU outputs) are often non negative or highly asymmetric.

Weight Only and Activation Only vs Full Quantization

Weight Only Quantization

Only the model weights are quantized. Activations remain in high precision floating point formats like FP16 or BF16. We do this because activations vary a lot between inputs and quantizing them can hurt accuracy, whereas weights are static and much easier to quantize. QLoRA is an example of weight only quantization used during fine tuning.

Activation Quantization

Only activations are quantized. This is used in frameworks like SmoothQuant and the Nvidia Transformer Engine.

Full Quantization

We quantize both weights and activations to maximize memory savings and speed up matrix multiplication.

Quantization Aware Training and Post Training Quantization

Quantization Aware Training

Quantization Aware Training (QAT) simulates quantization during the forward pass of training. Even though the model still computes gradients and updates in floating point (like FP16 or FP32) during training, it uses fake quantized weights and activations to calculate the loss. This helps the model adapt to the quantization errors and rounding effects before the final model is compiled.

If you train on pure floating point weights and then quantize them afterwards (Post Training Quantization or PTQ), you often see a drop in accuracy. QAT exists to force the model to adapt to these errors during the training process, so it performs much better during integer inference.

Fake Quantization

  1. Take a floating point value $x$.
  2. Quantize it to an integer: $q = \text{round}(x/s + z)$.
  3. Immediately convert it back to a float: $x_{\text{fake}} = s \cdot (q - z)$.

For example, an input like $16.2$ might be quantized and then dequantized back to a float value like $16.0$. This allows the model to experience the precision loss of quantization while still using floating point tensors for the actual backpropagation.

Why Fake Quantization?

Fake quantization simulates the precision loss of integer formats during training. This forces the model to adjust its remaining parameters to compensate for the quantization noise.

The Backward Pass

Quantization (specifically the rounding step) is non differentiable because its derivative is zero almost everywhere and undefined at integers. To train through this, we use a Straight Through Estimator (STE). We simply ignore the rounding step during the backward pass and pass the gradient straight through as if the derivative were 1: $$\frac{\partial \hat{x}}{\partial x} \approx 1$$

Efficient QAT

Standard QAT is expensive. Fake quantizing every activation and weight and then training with STE is compute heavy and slows down training.

Efficient QAT techniques include:

  1. Selective fake quantization: Only quantizing the most sensitive layers (like attention projections or specific linear layers) that account for the bulk of the quantization error, while leaving other layers in higher precision.
  2. Delayed activation quantization: Starting the training process with standard floating point activations and introducing fake quantization only in the later epochs.
  3. Range calibration: Statically calibrating the ranges before starting QAT to stabilize the scales.

The core technique is the same; you are just being smart about where and when you apply it.

Block Wise Training of All Parameters (Block AP)

Instead of quantizing the entire model at once, the model is split into blocks (like individual transformer layers).

For each block:

  1. We freeze all other blocks.
  2. In the active block, we replace the floating point weights $W$ with fake quantized weights $\widetilde{W}$ during the forward pass.
  3. During training, we update the weights using STE and also update the quantization parameters (like the scale $s$ and zero point).
  4. The scale and other parameters are updated during each training cycle to optimize the quantization range.
  5. After a set number of steps, we freeze the current block's weights and move to the next block.

The loss function used is: $$\mathcal{L} = \mathcal{L}_{\text{task}}(f(\widetilde{W})) + \lambda \mathcal{R}(W,s)$$ where:

  • $\mathcal{L}_{\text{task}}$ is the task loss (like cross entropy or MSE).
  • $\mathcal{R}(W,s) = \|W - \widetilde{W}\|_2^2$ is the reconstruction loss (regularizer) that measures the distance between the unquantized weights $W$ and the quantized weights $\widetilde{W}$.
  • $\lambda$ is a hyperparameter that controls how strongly we force the floating point weights to stay close to their quantized representations.

End to End Quantization Parameter Tuning

After all blocks have been trained individually, we freeze the weights. We then run a final tuning pass over a small validation dataset, training only the quantization parameters (like scales) for the entire model. We do this to ensure global consistency across all blocks, finding a global scale configuration that will be used during inference.

Inference vs Training

Training

During training, we need high precision floating point representations. Gradients and weight updates require tiny, precise adjustments that integer math cannot represent. To prepare the model for inference, we use fake quantization (quantizing and dequantizing) during training so the model learns to tolerate the rounding errors. After training is complete, we convert and store the weights as integers.

Inference

During inference, we run matrix multiplications using the stored integers because integer math is much faster and more memory efficient. The scale is applied after the dot products are calculated.

We convert the results back to the floating point domain after heavy matrix multiplications because succeeding operations (like bias addition, softmax, LayerNorm, and residual connections) still require floating point precision.

Specifically:

  1. Store weights as integers after training.
  2. Store the scales and zero points (obtained after end to end tuning).
  3. Perform integer matrix multiplication.
  4. Multiply the result by the scale to convert back to the floating point domain for operations like softmax, LayerNorm, and residual connections.

Learned Step Size Quantization (LSQ)

Learned Step Size Quantization is a method where the scale (step size) that maps floating point values to integers is not computed statically from minimum/maximum values. Instead, it becomes a learnable parameter trained directly via backpropagation.

In standard quantization, we use a fixed scale: $$\hat{x} = s \cdot \text{clip}\left( \left\lfloor \frac{x}{s} \right\rceil, q_{\min}, q_{\max} \right)$$ In LSQ, the scale $s$ is treated as a learnable parameter with its own gradients. This allows the quantization grid to adapt to the weight distribution instead of using rigid minimum/maximum rules.

The step by step calculation is:

  1. Scale the value: $$v = \frac{x}{s}$$
  2. Round: $$q = \lfloor v \rceil$$
  3. Clip: $$a = \text{clip}(q, q_{\min}, q_{\max})$$
  4. Dequantize: $$\hat{x} = s \cdot a$$

This lets the quantization grid shift to where the weights actually lie, handling outliers much better than standard minimum/maximum quantization and yielding higher accuracy.

LSQ Gradient Computation

To update the scale parameter via backpropagation, we compute the gradient of the dequantized output $\hat{x}$ with respect to the scale $s$: $$\frac{\partial \hat{x}}{\partial s} = \begin{cases} q_{\min} & \text{if } x/s < q_{\min} \\ q_{\max} & \text{if } x/s > q_{\max} \\ \lfloor x/s \rceil - \frac{x}{s} & \text{otherwise} \end{cases}$$ In the unclipped region, this is the quantization error ($q - x/s$). The scale parameter adjusts itself to minimize this difference, reducing overall quantization error.

To stabilize training, LSQ normalizes the gradient of the scale parameter: $$g_s = \frac{1}{\sqrt{N \cdot q_{\max}}}$$ where $N$ is the number of elements in the tensor being quantized, and $q_{\max}$ is the upper integer bound of the quantization range. Normalization prevents the scale gradients from exploding when dealing with large tensors.

The scale is optimized indirectly through the standard training objective. Using the chain rule: $$\frac{\partial \mathcal{L}}{\partial s} = \frac{\partial \mathcal{L}}{\partial \hat{x}} \frac{\partial \hat{x}}{\partial s}$$ In the unclipped region: $$\frac{\partial \mathcal{L}}{\partial s} = \frac{\partial \mathcal{L}}{\partial \hat{x}} \left( q - \frac{x}{s} \right)$$ This means the scale learns to minimize the task loss rather than just the raw quantization error.

LSQ+

While standard LSQ only learns the scale, LSQ+ learns both the scale and the zero point.

The quantized value is computed as: $$q = \text{clip}\left( \left\lfloor \frac{x}{s} + z \right\rceil, q_{\min}, q_{\max} \right)$$ and the dequantized value is: $$\hat{x} = s \cdot (q - z)$$

The zero point shifts the quantization grid to fit the data distribution. The derivative of the dequantized output with respect to the zero point $z$ is: $$\frac{\partial \hat{x}}{\partial z} = \begin{cases} -s & \text{if } x/s + z < q_{\min} \\ 0 & \text{if } q_{\min} \le x/s + z \le q_{\max} \\ -s & \text{if } x/s + z > q_{\max} \end{cases}$$

Inside the clipping bounds, the zero point does not affect the output error directly, so the gradient is 0. Outside the bounds where values get clipped, the gradient is $-s$.

The zero point affects where the clipping window sits:

  • A positive gradient pushes the zero point to the right.
  • A negative gradient pushes the zero point to the left.

This allows all the activations to "vote" on where the clipping window should sit:

  • If many values are clipped on the left, the zero point moves left.
  • If many values are clipped on the right, the zero point moves right.
  • If values are well centered within the bounds, the gradient is 0.

LSQ+ also improves on the initialization of the scale parameter. While LSQ uses a simple initialization, LSQ+ initializes it using: $$s = \frac{\text{mean}(|x|)}{\sqrt{q_{\max}}}$$ This reduces the training warmup time and aligns the quantization grid properly with the weight distribution.

GPTQ

GPTQ (Generalized Post Training Quantization) is a post training weight only quantization algorithm. It is a second order algorithm that performs block wise quantization and applies error compensation to the remaining unquantized weights after each weight is quantized. It achieves less than 1% accuracy loss, which makes it a widely used technique for LLM inference.

GPTQ uses a Hessian based metric. When we quantize a weight matrix $W$ to $\widetilde{W}$, the goal is to minimize the output difference over a dataset of input activations $X$: $$\|WX - \widetilde{W}X\|_2^2$$ This is the difference in output activations, not the raw weight difference.

We can optimize each row of $W$ independently. For a single row $w$, the change from $w$ to the quantized row $\widetilde{w}$ is $\Delta w = \widetilde{w} - w$. The change in loss is approximated using a second order Taylor expansion: $$\delta \mathcal{L} \approx \frac{1}{2} \Delta w^T H \Delta w$$ where $H$ is the Hessian matrix of the squared error loss. GPTQ approximates the Hessian as twice the empirical covariance of the input activations: $$H = 2 X X^T$$ (or $2 X^T X$ depending on whether inputs are columns or rows). Solving the discrete optimization problem directly is computationally hard, so we use this quadratic approximation to find a cheap estimate of how the loss changes when we quantize weights.

This helps us figure out: if we quantize a weight $w_k$, what is the optimal way to adjust the remaining unquantized weights so that the overall output error stays as small as possible? The Taylor expansion tells us how the loss changes with respect to the weights via the Hessian matrix.

The Hessian

The Hessian matrix with respect to a row of $W$ is: $$H = 2 \mathbb{E}[x x^T]$$ which is twice the empirical covariance of the input activations. This covariance matrix shows how the different input dimensions are correlated, helping us identify which weight directions matter more for the output error.

The Residual

The residual vector $r$ represents the difference between the original weights $w$ and the quantized weights $q$: $$r = w - q$$ The quantity we want to minimize is the output error: $$\mathcal{L}(q) = \|X(w-q)\|_2^2 = r^T H r$$ Here, $r$ measures the quantization error in the weight space, while $H$ scales it to reflect the impact on the final output.

Error Compensation

When we quantize a single weight $w_k$, we want to adjust the remaining unquantized weights to compensate for the error and minimize the output difference.

To do this, we partition the residual vector $r$ and the Hessian matrix $H$ to isolate the $k$-th weight: $$r = \begin{bmatrix} r_k \\ r_{-k} \end{bmatrix}, \qquad H = \begin{bmatrix} H_{kk} & H_{k,-k} \\ H_{-k,k} & H_{-k,-k} \end{bmatrix}$$ where $r_k$ is the residual of the weight we are currently quantizing, and $r_{-k}$ represents the residuals of the remaining unquantized weights.

We can expand the quadratic loss as: $$\mathcal{L} = r^T H r = H_{kk} r_k^2 + 2 r_k H_{k,-k} r_{-k} + r_{-k}^T H_{-k,-k} r_{-k}$$ Since we have already quantized weight $k$, its residual $r_k$ is fixed. To find the optimal compensation for the remaining weights, we take the gradient of the loss with respect to $r_{-k}$ and set it to zero: $$\frac{\partial \mathcal{L}}{\partial r_{-k}} = 2 H_{-k,-k} r_{-k} + 2 H_{-k,k} r_k = 0$$ Solving for $r_{-k}$ gives: $$r_{-k} = -H_{-k,-k}^{-1} H_{-k,k} r_k$$

Efficient Computation of the Hessian Inverse

Recomputing the inverse of the submatrix $H_{-k,-k}$ for every single weight we quantize is extremely expensive. Instead, we compute the full Hessian inverse $G = H^{-1}$ once at the start. Using block matrix inversion properties, we can write the term $H_{-k,-k}^{-1} H_{-k,k}$ directly using the elements of $G$: $$H_{-k,-k}^{-1} H_{-k,k} = \frac{G_{-k,k}}{G_{kk}}$$ This simplifies the compensation update for the remaining weights to: $$\boxed{w_{-k} \leftarrow w_{-k} - \frac{G_{-k,k}}{G_{kk}} r_k}$$ where $G_{-k,k}$ is the $k$-th column of $G$ (excluding the $k$-th row) and $G_{kk}$ is the $k$-th diagonal element.

After quantizing weight $k$, we must update the inverse Hessian matrix $G$ to reflect the removal of the $k$-th weight. We do this efficiently using a step of Gaussian elimination on the inverse matrix: $$\boxed{G \leftarrow G_{-k,-k} - \frac{G_{-k,k} G_{k,-k}}{G_{kk}}}$$

The Complete GPTQ Workflow

For each block or layer in the model:

  1. Compute the Hessian matrix $H = 2 X X^T$ and its inverse $G = H^{-1}$.
  2. Loop through the weights one by one (or in columns): a. Quantize the current weight $w_k$ to get $q_k$. b. Compute the residual: $r_k = w_k - q_k$. c. Read the column $G_{-k,k}$ and the diagonal element $G_{kk}$ from $G$. d. Compute the compensation for the remaining unquantized weights: $$\Delta w_{-k} = -\frac{G_{-k,k}}{G_{kk}} r_k$$ e. Update the remaining floating point weights: $$w_{-k} \leftarrow w_{-k} + \Delta w_{-k}$$ f. Update the Hessian inverse matrix $G$ to remove the $k$-th coordinate: $$G \leftarrow G_{-k,-k} - \frac{G_{-k,k} G_{k,-k}}{G_{kk}}$$
  3. Record the final quantized weights and repeat for all layers.

By doing this channel by channel or block by block, GPTQ lets us squeeze massive models down to 4 bit weights with almost zero loss in accuracy. It is a neat trick that makes running local LLMs actually feasible on consumer hardware. Next up, we will look at how this compares to other activation aware quantization methods.
