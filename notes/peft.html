PEFT

Parameter Efficient Fine Tuning is a method used for fine tuning large language models by updating a small subset of the model's parameters while keeping the majority of the pretrained weights frozen.

This is preferred due to:

  1. Less computational cost
  2. Less training time
  3. Less storage

Intrinsic Dimensionality

paper

Intrinsic dimensionality of any model is the minimum number of dimensions required to fine tune a pretrained model on downstream NLP tasks.

According to this paper and concept, pretrained models are good enough that they already can perform whatever NLP task we want to throw at them. Fine tuning is only done to lead or direct the model into doing that task for specific queries or instructions.

This minimum dimensionality of fine tuning that is required to fine tune a model on a task is called intrinsic dimensionality. The paper shows that common NLP tasks within the context of pretrained representations have an intrinsic dimension several orders of magnitude less than the full parameterization.

Instead of optimizing the empirical loss in the original parameterization ($\theta^D$), the subspace method fine tunes the model via the following reparametrization in the lower dimensional $d$ dimensions: $$\theta^D = \theta^D_0 + P(\theta^d)$$ where $P : \mathbb{R}^d \to \mathbb{R}^D$ projects a parameter from a lower dimensional $d$ to the higher dimensional $D$.

Specifically, we can write: $$\theta^D = \theta^D_0 + M \theta^d$$ Here, we are showing that the full parameter update is represented by the pretrained weights plus the lower dimensional vector multiplied by the projection matrix. $M \in \mathbb{R}^{D \times d}$ is a projection matrix which projects the $d$ dimensions of fine tuning updates to the $D$ space. This matrix takes the $d$ dimensional fine tuning vector and expands it to the $D$ dimensional model. Projection means mapping a space to another space, in this case mapping something linearly from a smaller space to a larger space.

The Projection Matrix

In practice, storing a dense $D \times d$ matrix is too expensive. Instead, $M$ is structured using a fast projection method like the Fastfood transform, which avoids explicitly storing the full matrix. It is constructed from four matrices:

Permutation Matrix $\Pi$ It randomly shuffles coordinates and prevents updates from affecting only a specific region of weights. It ensures that the subspace is uniformly mixed throughout the full parameter space.

Random Sign Diagonal Matrix $D_1$ Each diagonal entry is randomly chosen as $+1$ or $-1$. This random sign flip helps avoid structured correlations and spreads updates across dimensions.

Random Gaussian Diagonal Matrix $D_2$ It scales each coordinate randomly using a Gaussian distribution, making the subspace projection random.

Hadamard Matrix $H$ This is a symmetric orthogonal matrix with entries $+1$ or $-1$. It enables the Fast Walsh Hadamard Transform, which reduces the matrix multiplication complexity from $O(D^2)$ to $O(D \log D)$. This allows fast multiplication.

Structure Aware Intrinsic Dimension

The basic projection method ignores the fact that some transformer layers are more important than others. Ideally, projection should happen in a way that gives more important weights a larger impact.

To fix this, we use Structure Aware Intrinsic Dimension (SAID), which modifies the formula: $$\theta_{D, i} = \theta_{D_0, i} + \lambda_i \cdot P(\theta_{d-m})_i$$ Here, $\lambda_i$ is a learned scalar parameter for each layer or parameter group $i$ that scales its update. Out of our total budget of $d$ dimensions, we reserve $m$ dimensions to learn these scale factors, leaving $d - m$ dimensions for the subspace parameter vector $\theta_{d-m}$. This lets the model automatically decide which layers need bigger updates.

Why is this important?

Intrinsic dimensionality is the theoretical foundation of parameter efficient fine tuning. Most of these methods rely on low rank matrices or low rank linear projections.

Because pretrained models already understand language and contain latent knowledge for most tasks, fine tuning is not about teaching the model new concepts from scratch. Instead, it is just directing the model to format and apply its existing knowledge to specific instructions. This is why updating a tiny fraction of the parameters works so well.

BitFit

Bias Term Fine Tuning does exactly what it says: it freezes all weight matrices and only fine tunes the bias terms.

In a typical model, bias parameters make up only about 0.05% to 0.09% of the total parameter count. For example, in the attention queries, keys, and values: $$Q^{m,l}(x) = W_{q}^{m,l} x + b_{q}^{m,l}$$ $$K^{m,l}(x) = W_{k}^{m,l} x + b_{k}^{m,l}$$ $$V^{m,l}(x) = W_{v}^{m,l} x + b_{v}^{m,l}$$

Because of this, BitFit is an incredibly cheap alternative to full fine tuning.

What BitFit does:

  1. Freeze all weight matrices.
  2. Keep biases trainable.
  3. Fine tune on a downstream task using standard gradient based updates on the bias terms.

Results show that on standard NLP tasks, BitFit achieves performance comparable to full fine tuning despite updating next to nothing. This supports the idea that pretrained models already contain the necessary representations, and we just need to shift the activations slightly via the biases to expose the right behavior.

Adapter Tuning

Adapters

Adapters are small trainable neural networks inserted into each transformer block while keeping the original pretrained model weights frozen.

While BitFit only updates about 0.05% to 0.1% of the parameters (which is sometimes not enough for complex tasks), adapters usually update 1% to 5% of the parameters. They do not edit the existing weights directly, but instead introduce new trainable layers.

Structure

An adapter block consists of a bottleneck structure with a down projection and an up projection linear layer. It is typically inserted after the multi head attention layer and after the feed forward layer.

Down Projection Let $h'$ be the input hidden state to the adapter block. The down projection is defined as: $$h_{\text{down}} = W_{\text{down}} h' + b_{\text{down}}$$ Here, $W_{\text{down}} \in \mathbb{R}^{r \times d}$ projects the hidden state from the model dimension $d$ down to a bottleneck dimension $r$ (where $r \ll d$).

Activation We apply a nonlinear activation function $\sigma$ (such as ReLU or GeLU) to learn complex features: $$h_{\text{mid}} = \sigma(h_{\text{down}})$$

Up Projection The up projection maps the hidden state back to the original model dimension $d$: $$h_{\text{up}} = W_{\text{up}} h_{\text{mid}} + b_{\text{up}}$$ where $W_{\text{up}} \in \mathbb{R}^{d \times r}$.

Finally, the adapter output is added back to the original input via a residual connection: $$h_{\text{out}} = h' + h_{\text{up}}$$

Often, a scaling factor $\alpha$ is used to control the influence of the adapter: $$h_{\text{next}} = h_{\text{residual}} + \alpha \cdot W_{\text{up}} \sigma(W_{\text{down}} h' + b_{\text{down}}) + b_{\text{up}}$$

During Inference

The adapter layers are computed during the forward pass. The correction they apply is small but precisely tuned for the downstream task, acting as a subtle steering force.

Prompt Tuning

Prompt tuning is a technique that involves modifying the input to a pretrained language model rather than altering the model parameters. Instead of manually designing discrete text prompts, we prepend a small set of trainable continuous parameters called soft prompts to the input sequence.

The training process is similar to a normal neural network, but the gradients only update these soft prompt embeddings.

The soft prompt embeddings are prepended to the input sequence: $$[\underbrace{v_1, v_2, \dots, v_k}_{\text{trainable prompt tokens}},\, x_1, x_2, \dots, x_n]$$

Specifically, prompt tuning introduces a parameter matrix: $$P \in \mathbb{R}^{k \times d}$$ where $k$ is the prompt length and $d$ is the model embedding dimension.

If $H \in \mathbb{R}^{n \times d}$ is the embedding matrix of the original input tokens, the final input sequence given to the model is: $$\tilde{H} = \begin{bmatrix} P \\ H \end{bmatrix} \in \mathbb{R}^{(k + n) \times d}$$

Large language models (typically larger than 10 billion parameters, as prompt tuning does not perform as well on smaller models) already have a variety of latent behaviors like text classification, translation, and information extraction. Prompt tuning coaxes out these behaviors by learning soft instructions that adjust how the self attention layers process the input. Think of it as teaching the model to think differently before producing an answer. The key is that transformers are extremely sensitive to changes in input embeddings.

Prefix Tuning

While prompt tuning only appends trainable embeddings at the input layer, their steering effect can diminish as we go deeper into the network. Prefix tuning solves this by prepending trainable prefix key and value vectors directly to the keys and values in the self attention mechanism at every single layer.

For each layer $l$, we introduce trainable prefix key and value matrices: $$P_K^{(l)} \in \mathbb{R}^{m \times d_k}, \quad P_V^{(l)} \in \mathbb{R}^{m \times d_v}$$ where $m$ is the prefix length and $d_k, d_v$ are the key and value dimensions.

These prefix matrices are concatenated to the original key and value matrices $K, V \in \mathbb{R}^{n \times d_k}$ (where $n$ is the sequence length): $$K' = \begin{bmatrix} P_K^{(l)} \\ K \end{bmatrix} \in \mathbb{R}^{(m + n) \times d_k}, \qquad V' = \begin{bmatrix} P_V^{(l)} \\ V \end{bmatrix} \in \mathbb{R}^{(m + n) \times d_v}$$

The attention calculation then proceeds normally, but uses these extended key and value matrices: $$\text{Attn}(Q, K', V') = \text{softmax}\left(\frac{Q {K'}^\top}{\sqrt{d_k}}\right) V'$$ This allows the prefix keys and values to act as virtual context tokens at every layer.

To stabilize training, the prefix vectors are not optimized directly as raw parameters. Instead, they are generated by a Multi Layer Perceptron (MLP) prefix encoder: $$p_i = \text{MLP}(z_i)$$ where $z_i$ is a trainable virtual vector. The MLP outputs a vector of size $d_{\text{prefix}} = d_k + d_v$, which is then split: $$p_i[1:d_k] \rightarrow P_{K,i}$$ $$p_i[d_k : d_k + d_v] \rightarrow P_{V,i}$$

Once training is complete, the MLP weights can be discarded, and we only need to keep the computed $P_K^{(l)}$ and $P_V^{(l)}$ for inference.

How it works:

  1. Initialize a set of trainable prefix vectors $z_i$.
  2. Feed them through the MLP encoder during training.
  3. Split the MLP outputs into prefix key and value vectors.
  4. Concatenate these prefix keys and values to the original keys and values of the transformer at each layer.
  5. Train only the prefix vectors $z_i$ and the MLP encoder weights while keeping the rest of the model frozen.

P Tuning

While prompt tuning learns prompt embeddings directly, they can be unstructured and unstable, especially during training on smaller models. P Tuning fixes this by using a prompt encoder (a small neural network consisting of a BiLSTM and an MLP) to generate context aware, structured prompt embeddings.

Architecture

We define virtual prompt tokens that act as index placeholders: $$[P_1, P_2, \dots, P_m]$$

Prompt Encoder Each virtual token is passed through the prompt encoder to produce the prompt embedding: $$E_p = f_\theta(P_1, \dots, P_m)$$ where $E_p \in \mathbb{R}^{m \times d}$ and $d$ is the hidden size of the language model. The encoder $f_\theta$ typically consists of an embedding layer, a BiLSTM, and an MLP. Passing the tokens through a BiLSTM ensures that the generated prompt embeddings are sequentially dependent and structured rather than completely independent vectors.

Once the input text tokens are converted to embeddings: $$E_x = E(X)$$ The final input sequence $\tilde{E}$ given to the transformer is a combination of the generated prompt embeddings $E_p$ and the input embeddings $E_x$ (either concatenated or interleaved based on the template): $$\tilde{E} = \text{concat}(E_p, E_x)$$

During training, only the prompt encoder parameters are updated, while the entire language model remains frozen. The prompt encoder learns to shape the input embeddings in a way that guides the frozen model to produce the target output.

P Tuning V2

In P Tuning v1, the prompt embeddings are only introduced at the input layer. In deep transformer models, their influence can fade by the time the activations reach the output layer.

P Tuning V2 solves this by adding trainable prompt embeddings at every single transformer layer.

Architecture

For each transformer layer $l \in \{1, \dots, L\}$, we define trainable prompt parameters: $$P^l \in \mathbb{R}^{m \times d}$$

Unlike P Tuning v1 or prefix tuning, P Tuning V2 discards the MLP or BiLSTM prompt encoder entirely. The authors found that when prompts are added to every layer, such reparameterization is no longer necessary. Instead, the prompt parameters $P^l$ are optimized directly.

These prompt parameters are concatenated directly to the keys and values of the self attention layers, exactly like prefix tuning. This allows the model to maintain task steering capacity across all layers without the computational overhead of a prompt encoder.

Low Rank Adaptation (LoRA)

Fine tuning a model normally means updating all parameters, which involves billions of weights. However, we do not need to change most of these weights independently. Pretrained models already have the general capability; we just need to shift their activation directions to guide them.

LoRA does this by freezing the original weights and adding a pair of low rank trainable matrices.

Formulation

Let the original weight matrix be $W \in \mathbb{R}^{d \times k}$, where $d$ is the output dimension and $k$ is the input dimension. We introduce two low rank matrices $A$ and $B$: $$A \in \mathbb{R}^{r \times k} \quad (\text{down projection})$$ $$B \in \mathbb{R}^{d \times r} \quad (\text{up projection})$$ where $r \ll \min(d, k)$ is the rank.

The total weight update $\Delta W$ is parameterized as: $$\Delta W = B A \in \mathbb{R}^{d \times k}$$

The modified forward pass for an input $x$ is: $$h = W' x = (W + \Delta W) x = W x + B(A x)$$

During training, the base weight matrix $W$ is kept frozen, and gradients only flow into the low rank matrices $A$ and $B$.

Why Low Rank?

During fine tuning, the weight update matrix $\Delta W$ has a very low intrinsic rank. By factoring $\Delta W$ into the product of two smaller matrices $B$ and $A$, we drastically reduce the number of parameters we need to train.

Initialization

To ensure that training starts with the exact behavior of the base model, we initialize:

  • $A$ using a random Gaussian distribution.
  • $B$ to all zeros.

This ensures that initially, $\Delta W = B A = 0$, meaning the model's behavior is unchanged at step zero.

Scaling

LoRA scales the low rank update by a constant factor: $$\Delta W = \frac{\alpha}{r} B A$$ where $\alpha$ is a constant scaling hyperparameter. Using $\frac{\alpha}{r}$ helps stabilize training because when we change the rank $r$, we do not need to re tune the learning rate or other hyperparameters.

LoRA is typically applied to the attention projection matrices, such as the query matrix $W_Q$ and the value matrix $W_V$, though it can also be used in feed forward layers. Updating the query and value projections usually yields the best performance.

Inference Modes

LoRA supports two different ways of handling inference:

Merged Inference Before deploying, we compute the combined weights: $$W_{\text{final}} = W + \frac{\alpha}{r} B A$$ We can then discard the matrices $A$ and $B$ and use $W_{\text{final}}$ directly. This adds zero latency during inference.

Unmerged Inference We keep the base weights and LoRA matrices separate: $$h = W x + \frac{\alpha}{r} B(A x)$$ This adds a tiny bit of latency, but it allows us to hot swap different LoRA weights on the fly (e.g., swapping between a summarization LoRA, a Q&A LoRA, and a coding LoRA) using the same frozen base model.

QLoRA

QLoRA reduces the VRAM footprint of fine tuning by keeping the base model weights in a 4 bit quantized form (specifically using a format called NormalFloat 4, or NF4) while still allowing high quality fine tuning using LoRA.

Instead of storing pretrained weights in 16 bit form, QLoRA quantizes the base model: $$W_{\text{4bit}} = \text{quantize}(W)$$

To squeeze out even more memory, QLoRA uses double quantization, which quantizes the quantization constants themselves (e.g., from 32 bit floats to 8 bit floats), saving about 0.37 bits per parameter.

During the forward and backward passes, the 4 bit weights are dynamically dequantized back to 16 bit (BF16 or FP16) to perform the matrix multiplication: $$h = \text{dequantize}(W_{\text{4bit}}) x + \frac{\alpha}{r} B(A x)$$

The base weights stay in 4 bit in memory, while only the trainable LoRA parameters $A$ and $B$ and their gradients are stored and updated in 16 bit or 32 bit format.

Paged Optimizers

Optimizers like Adam or AdamW need to track the first moment $m$ and second moment $v$ for every trainable parameter. When fine tuning, these optimizer states can consume multiple times the memory of the trainable parameters themselves. Even with LoRA, training can cause sudden memory spikes due to activation memory, gradients, and temporary buffers during backpropagation.

When the GPU VRAM is completely full, even a small memory request will fail and throw an out of memory error.

What Paged Optimizers Do

Paged optimizers solve this by utilizing CUDA Unified Memory to perform page to page transfers between the GPU VRAM and CPU system RAM.

During training, when the GPU experiences memory spikes, CUDA automatically pages inactive optimizer states to the CPU RAM.

Once the backward pass is complete and the optimizer needs to perform the weight update, those states are paged back to the GPU VRAM.

How it works

Instead of allocating a single contiguous block of GPU memory for optimizer states like $m$ and $v$:

# Standard allocation (can easily fail on VRAM spikes)
m = tensor(size=[10000000], device="cuda")
v = tensor(size=[10000000], device="cuda")

Paged optimizers allocate them as paged tensors that can swap chunks (pages) to CPU memory when needed:

# Paged allocation utilizing CUDA Unified Memory
m = PagedTensor([
    page1: pointer to GPU (4KB)
    page2: paged to CPU RAM (4KB)
    page3: pointer to GPU (4KB)
    ...
])

QLoRA uses paged optimizers because even if we squeeze the base weights down to 4 bit, the gradients and optimizer states still take up standard floating point space. Offloading these optimizer states to CPU RAM is a lifesaver when VRAM is tight.

Overall, these parameter efficient fine tuning methods show that we do not need to rewrite the entire brain of a model just to get it to follow new instructions. By steering it with adapters, prompts, or low rank updates, we get the same performance with a fraction of the hardware headache.
