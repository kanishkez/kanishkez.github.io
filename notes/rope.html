RoPE

Rotary Position Embedding, or RoPE, has basically become the standard for modern LLMs like Llama. Before getting into how it actually works, we need to look at why older positional embeddings, like absolute, learned, or standard sinusoidal ones, fall short when context windows get huge.

The problems with sinusoidal, absolute, and learned positional embeddings

  1. They can represent relative positions (sinusoidal can) but the model doesn't understand that tokens 4 and 6 are closer than tokens 1 and 100 are.
  2. At huge sequence lengths, attention scale degrades. The dot product between two distant positions becomes noisy and unstructured, so attention also gets degraded.
  3. Phase drifting: Sinusoidal encoding uses multiple frequencies. Over long sequences, different frequencies create interference patterns. Mathematically: $\sin(\omega(t + \Delta)) \approx \sin(\omega t) \quad \text{when } \omega \Delta \approx 2k\pi$ So positions thousands of tokens apart can become phase aligned and indistinguishable.

So, Rotary position embeddings came to exist to encode positions in such a way that:

  1. Encodings represent relative positions in a way that the model understands them.
  2. They scale naturally to longer context.
  3. They integrate directly into attention math.

Rotation

Consider a two dimensional vector: $$x = \begin{bmatrix} x_1 \\ x_2 \end{bmatrix}$$ A counterclockwise rotation by angle $\theta$ is: $$R(\theta) = \begin{bmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{bmatrix}$$ So the rotated vector is: $$x' = R(\theta)x$$

Query and key vectors in attention have high dimensions. RoPE splits them into $d/2$ independent two dimensional rotation planes.

So for a query vector of dimension $d$, we reshape it into pairs: $$q = [(q_1,q_2), (q_3,q_4), \dots, (q_{d-1},q_d)]$$ By this, we mean that we are splitting the single token representation into two dimensional sub vectors.

For a single token $t$ and a single attention head, if the query vector is of length $d_{head}$, we split that into $d_{head}/2$ pairs of two dimensions. Each pair is rotated with a potentially different rotation angle.

Position Dependent Rotation

For a token at position $t$, each two dimensional pair gets a unique angle: $$\theta_{t,m} = t \cdot \omega_{m}$$ where:

  • $t$ is the token index ($0, 1, 2, \dots$)
  • $m$ is the pair index ($0$ to $d/2-1$) representing which pair it is
  • $\omega_m$ is the frequency assigned to that pair (how fast it rotates)

Frequency

The frequency of the rotation influences how fast or how much the rotation happens. In RoPE, the first dimensions rotate slowly, while higher dimensions rotate faster. This creates a multi frequency encoding similar to sinusoidal embeddings.

The frequencies used by RoPE are: $$\omega_{m} = 10000^{-2m/d}, \qquad m = 0, 1, \ldots, \frac{d}{2}-1$$ So for each pair of dimensions, we assign a different frequency. We can also write that as: $$\omega_{m} = \left(10000^{-\frac{2}{d}}\right)^{m}$$ In the paper, a slightly different notation might be used, but the formula is functionally the same: the absolute position index is multiplied by a frequency to get the angle of the rotation.

So the rotated query becomes: $$\tilde{q}_{t,(2m:2m+1)} = R(\theta_{t,m})\, q_{(2m:2m+1)}$$ For keys: $$\tilde{k}_{t,(2m:2m+1)} = R(\theta_{t,m})\, k_{(2m:2m+1)}$$

Integration into Attention Score

The attention score is: $$\text{score}(i,j) = \tilde{q}_i^\top \tilde{k}_j$$

Relative Positioning

Our main goal in using RoPE is to ensure that the positions of the query and key vectors affect their score. RoPE achieves this by using their absolute position difference, which is how far the query and key are from each other in the input sequence.

Here is how that works mathematically. Because rotation matrices are orthogonal, we have: $$R(\theta_i)^\top R(\theta_j) = R(\theta_j - \theta_i)$$ So we get: $$\tilde{q}_i^\top \tilde{k}_j = q_i^\top R(\theta_j - \theta_i)k_j$$ Since $\theta_{t,m} = t\omega_m$, for a specific pair we get: $$\theta_j - \theta_i = (j-i)\omega_m$$ This means that the attention score between query $i$ and key $j$ is directly influenced by the relative distance between them ($j-i$). So the attention score depends on how far apart the tokens are, not where they are globally in the sequence.

Through this mechanism, RoPE turns absolute position into relative position awareness.

Absolute Position

Since the rotation is still a function of the absolute position $t$, the representations are still influenced both by the relative position between $q$ and $k$ and the absolute position of the token itself.

Rotation Matrix

RoPE applies a separate two dimensional rotation to every adjacent pair of dimensions of the embedding. So, we use a block diagonal matrix where each block rotates a different two dimensional slice of the vector. This matrix represents all the rotations combined.

Each two by two block is a standard rotation matrix: $$R(\theta_i t) = \begin{pmatrix} \cos(t\theta_i) & -\sin(t\theta_i) \\ \sin(t\theta_i) & \cos(t\theta_i) \end{pmatrix}$$ So the full rotation matrix for dimension $d$ is: $$R_{d,\Theta,t} = \begin{pmatrix} R(\theta_1 t) & 0 & 0 & \cdots & 0\\ 0 & R(\theta_2 t) & 0 & \cdots & 0\\ 0 & 0 & R(\theta_3 t) & \cdots & 0\\ \vdots & \vdots & \vdots & \ddots & 0\\ 0 & 0 & 0 & \cdots & R(\theta_{d/2} t) \end{pmatrix}$$ But performing a full matrix multiplication like this is computationally wasteful because it scales as $O(d^2)$.

Efficient Implementation using Complex Numbers

Here is where complex numbers come in. Instead of treating each pair as a two dimensional vector, we can interpret it as a complex number: $$z_i = q_{2i} + j q_{2i+1}$$ So the rotation becomes a simple multiplication: $$z'_i = z_i \cdot e^{j \, t\theta_i}$$ Because of Euler's formula: $$e^{j\alpha} = \cos\alpha + j\sin\alpha$$ A single complex multiplication by $e^{j t\theta_i}$ performs the exact same rotation. This means we do not need to use matrices; we can just perform elementwise operations using complex numbers.

The full implementation steps:

  1. Precompute frequencies: $$\theta_{i} = 10000^{-2i/d}, \quad i = 0, 1, \dots, \frac{d}{2}-1$$
  2. Compute the phase per position: $$\phi_i(t) = t\theta_i$$
  3. Form the rotation factor: $$e^{j\phi_i(t)} = \cos(\phi_i(t)) + j \sin(\phi_i(t))$$
  4. Apply the rotation: $$q_{t,i}' = q_{t,i} e^{j\phi_i(t)}, \quad k_{t,i}' = k_{t,i} e^{j\phi_i(t)}$$ where $q_{t,i}$ and $k_{t,i}$ are the complex representation of the $i$-th pair at position $t$.

Real Number Implementation in Code

In actual code, we often implement this using real numbers to keep it fast and compatible with frameworks that do not have great complex number support.

To compute the rotation for a pair $(x, y)$: $$x' = x\cos\theta - y\sin\theta$$ $$y' = x\sin\theta + y\cos\theta$$ We can write this in a vectorized way. If we take the pair $(x, y)$ and rotate it by 90 degrees (which is equivalent to multiplying by the imaginary unit $j$): $$(x, y) \rightarrow (-y, x)$$ Then the rotated pair is computed as: $$(x', y') = (x, y) \cos\theta + (-y, x) \sin\theta$$ This matches the rotation formula exactly.

In code, we construct a rotated version of the query/key vector (often called rotate_half or rotate_every_two) where we swap and negate the pairs. Then we apply the cosine and sine coefficients elementwise: $$\tilde{q} = q \odot \cos(\theta) + \text{rotate}(q) \odot \sin(\theta)$$ This avoids any matrix multiplication.

Finally, the attention score calculation is: $$A_{ij} = \frac{\tilde{q}_i \tilde{k}_j^\top}{\sqrt{d}}$$ where $\tilde{q}_i$ and $\tilde{k}_j$ are the rotated query and key vectors.

So, at the end of the day, RoPE gives us the best of both worlds. We get the absolute position encoding's simplicity in the implementation, but we mathematically preserve relative distance between tokens. That is why it scales so well and has become the default choice for modern LLMs.
