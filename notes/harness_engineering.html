What is a Harness?

Essentially, a harness is everything we provide to a model, such as tools, tests, .md files, instructions, sandboxes to operate in etc.

Whatever set of tools, memory , instructions and environment we provide to the model to make it act better as an agent, actually helps the model performance.

For example, OpenAI team worked on building a project solely using Codex. At first, it was difficult but after they harness engineered, they were actually able to build a codebase of millions of lines just by codex.

It starts with the very basics: an Agents.md file that tells the model what tech stack to use, how to verify and run tests etc, what architecture to use etc.

An Agents.md file can significantly increase model's capabilities.

So, model capability and execution are two different things, a good harness is very important to ensure that any model works well.

Think about it this way: If you're a new engineer given access to just a codebase without any prior context, theres not much you can understand even if you're a really good coder. It works the same with agents as well.

Everything you provide as context and infrastructure to the model determines how much of the model's capability gets realized.

Claude Code reads everything from a Claude.md repo so that it understands everything.

Cursor uses .cursorrules file.

A harness should be a map, not a manual. A good harness uses rule to constraint the agent, rather than just listing a bunch of instructions.

5 subsystem Harness model:

Instructions: Agents.md or Claude.md files that contain a project overview, architecture and tech stack instructions.

Tool subsystem: Shell tools, execution tools and access to a file subsystem.

Progress/State subsystem: Ensure it can track everything that has been going on through a simple progress.md or git commits.

Runtime/Environment subsystem: Mention packages to use via package.json or pyproject.toml to specify runtime versions and use Docker to make the environment reproducible.

Feedback/Results subsystem: This is very important, list verification commands in Agents.md or any other file of your choice. Mention tests and full verification methods to make sure everything is working.

Tests: pytest tests/ -x

All 5 of these components makeup a good harness.

A good repo structure

It is important to make sure that everything about the code lives inside the codebase, every small instruction, every architecture decision should live inside the codebase. The codebase should be so fool-proof that any new agent in a new session should be able to look at it and understand everything.

A good repo structure looks like:

project/
├── AGENTS.md              # Entry: project overview, run commands, hard constraints
├── src/
│   ├── api/
│   │   ├── ARCHITECTURE.md  # API layer architecture decisions
│   │   └── ...
│   ├── db/
│   │   ├── CONSTRAINTS.md   # Database operation hard constraints
│   │   └── ...
│   └── ...
├── PROGRESS.md             # Current progress: done, in-progress, blocked
└── Makefile                # Standardized commands: setup, test, lint, check

Splitting Instructions

After making a good Agents.md file, if you adding hard enforced rules or keep increasing the size of the file, there will come a point where the model performance will deteriorate again.

A model has a limited context window, a rule buried in 1000 lines of instructions is less likely to be seen.

'Lost in the middle' paper states that LLMs utilize information in the middle of long texts/files less effectively than at the beginning or end.

So, imagine a very important constraint that is buried in 1000 lines of rules, the agent will just ignore it.

Important things to ensure:

Agents.md file should be at 50-200 lines and should only contain the essentials: a project overview, tests and hard constraints.

# AGENTS.md

## Project Overview
Python 3.11 FastAPI backend, PostgreSQL 15 database.

## Quick Start
- Install: `make setup`
- Test: `make test`
- Full verification: `make check`

## Hard Constraints
- All APIs must use OAuth 2.0 authentication
- All database queries must use SQLAlchemy 2.0 syntax
- All PRs must pass pytest + mypy --strict + ruff check

## Topic Docs
- API Design Patterns (`docs/api-patterns.md`) — Required reading when adding endpoints
- Database Rules (`docs/database-rules.md`) — Required when modifying database operations
- Testing Standards (`docs/testing-standards.md`) — Reference when writing tests

Each specific topic file can be organized by subject in a docs directory, an agent only reads them when needed.

Every instruction should have an example as to why it was added.

Working with Context Windows

Context windows are finite. No matter the window size, long tasks will eventually exhaust them. Anthropic observed that agents exhibit rushed finish behavior when approaching context limits, ending tasks early to avoid information loss. At its core, it's an irrational resource anxiety. This is context anxiety.

So, ultimately you will run out of context over long horizon tasks.

This is how to handle it: State persistence files, files that let a new session resume where the last one left off. This includes creating progress files, next action files and verification records.

Progress.md: It just records the progress of the current session.

# Project Progress

## Current State
- Latest commit: abc1234 (feat: add user preferences endpoint)
- Test status: 42/43 passing (test_pagination_edge_case failing)
- Lint: passing

## Completed
- [x] User model and database migration
- [x] Basic CRUD endpoints
- [x] Auth middleware integration

## In Progress
- [ ] Pagination feature (90% - edge case test failing)

## Known Issues
- test_pagination_edge_case returns 500 on empty result sets
- Need to confirm whether deleted users should appear in listings

## Next Steps
1. Fix pagination edge case bug
2. Add "include deleted users" query parameter
3. Update API documentation

Decisions.md: Records important decisions architecture and design wise. Also contains the reasons for the decisions taken.

# Design Decisions

## 2024 01 15: Use Redis for user preferences caching
- Reason: High read frequency (every API call), small data size
- Rejected alternative: PostgreSQL materialized view (high change frequency makes maintenance cost not worthwhile)
- Constraint: Cache TTL of 5 minutes, active invalidation on write

Using git commits as checkpooints.

Explain the model everything it needs to do after a new session starts. This can just be placed in Agents.md such as:

## At session start (clock in)
1. Read PROGRESS.md for current state
2. Read DECISIONS.md for important decisions
3. Run make check to confirm repo is in consistent state
4. Continue from PROGRESS.md "Next Steps" section

## Before session end (clock out)
1. Update PROGRESS.md
2. Run make check to confirm consistent state
3. Commit all completed work

Initializing a new session

  1. Make sure that when a new session starts, the agent's environment has no issues, all dependancies are installed.
  2. Make sure that all example test cases and verifications work.
  3. Get a checklist that has to be cleared everytime you initialize a new session.

How to get Tasks done

Completing a task Make sure that you split the entire work into an ordered task list, explain each task and what is the 'acceptance' criteria.

# Task Breakdown

## Task 1: User Authentication Basics
- Implement JWT auth middleware
- Add login/register endpoints
- Acceptance: pytest tests/test_auth.py all passing

## Task 2: User Profile Page
- Implement user profile CRUD
- Add profile edit form
- Acceptance: pytest tests/test_profile.py all passing

## Task 3: Search Feature

More often that not, there will be two problems you will encounter when you ask your agent to add a feature: Overreach and Under-finish.

Overreach refers to when the agent takes on more tasks than optimal and doesn't finish any of them. Under-finish happens when an agent assumes that a task is finished after completing only a small portion of it, delivering an incomplete solution.

To avoid these two issues:

Enforce Work In Progress rules: Enforce WIP=1 in your harness ( in agents.md ), so that the agent is only allowed to work on one task at any time.

Define how a task can be marked as finished. There should be explicit verification tests done so that the feature is delivered.

Implementing Feature Lists

Typically, an agent can hugely benefit from a proper feature list. Feature lists need to be implemented in such a way that the agent understands when a feature is completed and also understand all the features that need to be added.

Each feature's description should contain: description, verification command and current state.

A feature list should contain:

  1. A scheduler that schedules which features to work on.
  2. A verifier that can verify a feature.
  3. A progress tracker.

Here is how to define a feature list format: 

{
  "id": "F03",
  "behavior": "POST /cart/items with {product_id, quantity} returns 201",
  "verification": "curl -X POST http://localhost:3000/api/cart/items -H 'Content-Type: application/json' -d '{\"product_id\":1,\"quantity\":2}' | jq .status == 201",
  "state": "passing",
  "evidence": "commit abc123, test output log"
}

Write rules in your Agents.md file:

## Feature List Rules
- Feature list file: /docs/features.md
- Only one feature active at a time
- Verification command must pass before marking as passing
- Don't modify feature list states yourself — the verification script updates them automatically

Tests and Verification

Agents tend to be overly-confident about task completions. They feel that the work is done even if reality is far from it.

Defining tests and verifications is therefore, very important to ensure that everything works.

To do this, we need to ensure that we have proper termination criteria. An agent should satisfy all conditions before completing execution of the task.

Verification should happen in 2 layers, the first layer checks if the code correctly implements what was asked and the second layer checks whether the feature/task or system meets the end-to-end requirements.

We need to verify functional correctness, then address performance and then handle style.

When an agent is asked to evaluate its work, it will always provide positive assessments ( according to Anthropic's research recently).

To solve this, we need to enforce how verification is done properly.

Termination Judgement: The harness should independently execute termination validation using set rules instead of just taking the agent's word for it. You can spell this out in your Agents.md:

## Definition of Done
- Feature complete = end-to-end verification passed, not "code is written"
- Required verification levels:
  1. Unit tests pass
  2. Integration tests pass
  3. End-to-end flow verification passes
- Do not proceed to level 2 if level 1 fails
- Do not proceed to level 3 if level 2 fails

Using a 3 layer termination Validation Layer 1: Ensures that the syntax and static code used is correct.

Layer 2: Executes the tests, checks application startup and validates if everything runs properly.

Layer 3: Tests everything properly according to how an actual user would. Real-world execution tests are done.

Providing proper feedback

It is useless to just say 'Test failed', it is important that the agent understands what went wrong and how it went wrong. So, providing specific actionable feedback to the agent and letting the agent self-correct itself is very important.

End to End testing and Verification

When an agent knows its work will be validated by end to end tests, its coding behavior changes. It focuses on architectural boundaries and ensures that everything works together.

Your harness should include multiple levels of tests as specified to ensure that everything works end to end.

Make sure that components, features work together well instead of features passing their own unit tests but failing to work together.

Tests should detect cross-component defects instead of isolating features.

Observability

Increasing the agent's observability is important so that it sees its own mistakes and understands why things aren't working.

This comes down to two things:

  1. Runtime observability includes logs, traces and events that tells the agent what happens during runtime and also what goes wrong.
  2. Process observability helps the agent understand decisions such as plans, acceptance criteria etc.

Enforcing Observability

Ensure that the harness collects the following signals/logs:

  1. Application lifestyle phases
  2. Feature path execution
  3. Data flow
  4. Resource utilization
  5. Errors and Exceptions

Create a scoring rubric that gives a quantifiable score about every feature, every task such as:


| Dimension | A | B | C | D |
|-----------|---|---|---|---|
| Code correctness | All tests pass | Main flow passes | Partial pass | Build fails |
| Architecture compliance | Fully compliant | Minor deviations | Obvious deviations | Serious violations |
| Test coverage | Main + edge cases | Main flow only | Only skeleton | No tests |

Runtime signals are important so that the agent understands what exactly happens.

Make sure that after every session, the state of the codebase is clean

This means that make sure everything like builds, tests, progress and properly checked.

Make sure you track everything that has been done and completed so that you can use it for the next session.

So, yes this was a guide to harness engineering and how to get the most out of your agents. Happy Coding!
