KV Caching

When we generate text with transformers, generating token by token is incredibly slow because we keep recalculating the same representations. KV caching is a standard optimization to speed up inference by saving previously computed key and value states. Here is a breakdown of how it works under the hood and why it is so necessary.

In self attention, every token wants to "look at" other tokens in the same sequence and decide how much attention to pay to them.

That decision is made using the dot product between Query and Key vectors.

For a given token $i$:

$$\text{attention weights for token } i = \text{softmax}\left( \frac{Q_i K^\top}{\sqrt{d_k}} \right)$$

where $Q_i$ is the query vector for token $i$ and $K$ is the matrix containing key vectors for all tokens processed so far. Then, using those weights, we combine the Values $V_j$ of all tokens to produce a new representation for token $i$:

$$\text{output}_i = \sum_j \text{attention weight}(i, j) \, V_j$$

When you process a new token, you take the old keys and value vectors to see what has been generated so far and the current token's query being generated.

So in each iteration of generating a new token, old key and value vectors are used.

In vanilla transformers, these vectors are computed again and again for each step:

  • For $t = 2 \rightarrow$ recompute token $1$
  • For $t = 3 \rightarrow$ recompute tokens $1$ and $2$
  • For $t = 4 \rightarrow$ recompute tokens $1, 2, 3$
  • For $t = n \rightarrow$ recompute tokens $1$ to $n - 1$

What this does is that it makes the decoder time complexity $O(n^2)$ for generating a sequence of length $n$.

Since: $$1 + 2 + 3 + \dots + n = \frac{n(n+1)}{2} = O(n^2)$$

Since this is obviously inefficient, we introduce KV caching.

What we do is that instead of recomputing key and value tokens, we store them in a tensor in the GPU's VRAM so that we just fetch those values instead of recomputing them again and again.

How KV Cache actually looks like

Inside the GPU they are stored in tensors like these:

  • K cache $\rightarrow$ (num layers, batch size, num heads, sequence length, head dimension)
  • V cache $\rightarrow$ (num layers, batch size, num heads, sequence length, head dimension)

Every time a new token is generated, we compute its key and value vectors and append them to the cache tensors.

The Trade Off

The big trade off is that as the model size increases, the size of the KV cache also increases. All of this is stored in GPU VRAM. So it might grow really big and clutter our VRAM.

So KV caching basically trades off GPU memory for inference speed. We save a ton of compute time by not reprojecting past tokens, but we pay for it with a massive memory footprint. Managing this memory overhead is where things like PagedAttention and other cache optimization techniques come into play.
