Retrieval Augmented Generation

RAG is a neat way to give a language model access to external, real time knowledge when it runs inference, rather than expecting it to memorize every single fact during training. We are basically feeding the model context on the fly so it can generate better, more grounded answers.

Instead of:

$$Answer = f_\theta(prompt)$$ we do:

$$Answer = f_\theta(\text{prompt + Retrieved Documents})$$

The key idea is:

Do not force the model to memorize all knowledge. Instead, retrieve relevant information at inference time

This idea was introduced in the paper:

  • Retrieval Augmented Generation for Knowledge Intensive NLP Tasks

Retrieval Types

Retrieval is broadly of two types:

  1. Sparse Retrieval
  2. Dense Retrieval

Modern systems often combine both using Hybrid Retrieval.

Sparse Retrieval

Sparse retrieval methods represent documents as high dimensional sparse vectors If the vocabulary size is V, the document representation is:

$$d \in \mathbb{R}^{V}$$

Since a document only contains a small number of words, most entries are zero, making the vector sparse.

Sparse retrieval relies on exact token matching. Example: Query:

machine learning

Document:

machine learning tutorial

This matches well because tokens overlap.

But: Query:

car

Document:

automobile

No match occurs.

TF IDF

TF IDF stands for Term Frequency and Inverse Document Frequency. It measures how important a word is to a document relative to the corpus.

Term Frequency

Term frequency measures how often a word appears in a document.

$$TF(t, d) = \frac{\text{count}(t \in d)}{|d|}$$ Where:

  • t = term
  • d = document
  • |d| = total words in document

This captures the idea that:

Words that appear frequently in a document are likely important for that document.

Inverse Document Frequency

Some words appear in many documents (like "the", "is"). Such words carry very little information. IDF measures how rare a term is across documents.

$$IDF(t) = \log \frac{N}{df(t)}$$

Where:

  • N = total number of documents
  • df(t) = number of documents containing term t Rare words receive higher weight.

TF IDF Weight

TF and IDF are combined:

$$w(t, d) = TF(t,d) \cdot IDF(t)$$

Thus a term is important if:

  • it appears frequently in a document
  • but rarely across the corpus

Cosine Similarity

Documents and queries are represented as vectors. Similarity is usually computed using cosine similarity:

$$\cos(q, d) = \frac{q \cdot d}{|q| |d|}$$

Where:

  • q = query vector
  • d = document vector

Cosine similarity measures the angle between vectors.

Key advantage:

It normalizes for document length.

Two documents with similar word distributions will have a high cosine similarity even if one is longer.

Limitations of TF IDF

  • No semantic understanding
  • Cannot handle synonyms
  • High dimensional vectors
  • Sensitive to vocabulary mismatch

BM25

BM25 is an improved ranking function derived from probabilistic information retrieval.

It improves TF IDF by introducing:

  1. term frequency saturation
  2. document length normalization

BM25 scoring function:

$$BM25(q, d) = \sum_{t \in q} IDF(t) \cdot \frac{f_{t,d}(k_1+1)} {f_{t,d} + k_1\left(1 - b + b\frac{|d|}{avgdl}\right)}$$

Where:

  • f_{t,d} = frequency of term t in document
  • |d| = document length
  • avgdl = average document length
  • k_1 = term saturation parameter
  • b = length normalization parameter Typical values:
$$k_1 \approx 1.2$$ $$b \approx 0.75$$

Modified IDF in BM25

BM25 uses a modified IDF:

$$IDF(t) = \log \left( \frac{N - df(t) + 0.5}{df(t) + 0.5} \right)$$

This prevents extreme values for rare words.

Why BM25 Works Better

  • Prevents frequent words from dominating
  • Penalizes long documents
  • Handles term frequency nonlinearly
  • More robust than TF IDF

Many search engines like:

  • Elasticsearch
  • Apache Lucene

use BM25 as the default ranking function.

Dense Retrieval

Dense retrieval represents documents using neural embeddings.

Instead of sparse vectors:

$$d \in \mathbb{R}^{|V|}$$

we use low dimensional dense vectors:

$$d \in \mathbb{R}^{768}$$

or

$$d \in \mathbb{R}^{1024}$$

These vectors are generated using neural networks.

Example embedding:

"machine learning is powerful"

might become

$$[0.14, -0.81, 0.42, …, 0.17]$$

Dense Retrieval Pipeline

  1. Encode documents
$$d = f_\theta(document)$$
  1. Encode query
$$q = f_\theta(query)$$
  1. Compute similarity

Usually:

Dot product

$$score(q,d) = q^T d$$

or cosine similarity.

  1. Retrieve nearest neighbors.

Training Dense Retrievers

Dense retrievers are trained using contrastive learning.

For each query:

  • one positive document
  • several negative documents

Loss function:

$$L = -\log \frac{e^{q \cdot d^+}} {\sum_{i=1}^{n} e^{q \cdot d_i}}$$ The model learns to:

  • pull relevant documents closer
  • push irrelevant ones away

Dense retrieval became popular with:

  • Dense Passage Retrieval for Open Domain Question Answering

Hybrid Retrieval

Modern retrieval systems combine sparse and dense retrieval.

Score is typically:

$$Score = \alpha \cdot BM25 + (1-\alpha) \cdot DenseScore$$

Why hybrid works better:

Sparse retrieval handles:

  • exact tokens
  • numbers
  • names
  • code

Dense retrieval handles:

  • semantics
  • paraphrases
  • long questions

Most production RAG systems use hybrid retrieval.

Vector Indexing

Dense retrieval requires searching through millions of embeddings.

Brute force search would require computing similarity with every vector.

Cost:

$$O(N)$$

This becomes too slow for large datasets. Vector indexing solves this problem.

FAISS

One of the most widely used libraries for vector search is:

  • FAISS

FAISS is a library designed for fast similarity search on dense vectors.

It allows efficient retrieval of nearest neighbors in large embedding datasets.

FAISS Indexing Workflow

Step 1: Generate embeddings

documents → embeddings

Matrix representation:

$$E \in \mathbb{R}^{n \times d}$$

Where:

  • n = number of documents
  • d = embedding dimension

Step 2: Create index

Example:

index = faiss.IndexFlatL2(d)

Step 3: Add vectors

index.add(embeddings)

The embeddings are stored internally.

Step 4: Query

D, I = index.search(query_vector, k)

Returns:

  • I = indices of nearest vectors
  • D = distances

Approximate Nearest Neighbor Search

To scale retrieval, FAISS implements ANN algorithms.

These approximate the nearest vectors much faster than brute force.

Common Index Types

Flat Index

Exact search.

Searches every vector.

Complexity:

$$O(N)$$

Accurate but slow.

IVF (Inverted File Index)

Vectors are first clustered using k means.

Example:

10M vectors → 1000 clusters

Each vector belongs to a cluster.

Query workflow:

  1. find closest clusters
  2. Search only within those clusters

This reduces the search space dramatically.

HNSW (Hierarchical Navigable Small World Graph)

Vectors are stored as nodes in a graph.

Edges connect similar vectors.

Search works by navigating the graph toward the nearest neighbor.

Complexity roughly:

$$O(\log N)$$

This is one of the fastest ANN methods.

Product Quantization (PQ)

Used to compress vectors.

Instead of storing full embeddings, vectors are approximated using quantized representations.

Benefits:

  • reduces memory usage
  • enables large scale indexing.

HNSW Indexing

HNSW (Hierarchical Navigable Small World Graph)

HNSW is one of the most widely used Approximate Nearest Neighbor (ANN) algorithms for dense vector retrieval.

It is used in systems like:

  • FAISS
  • Milvus
  • Weaviate

The idea is to represent embeddings as nodes in a graph, where edges connect similar vectors.

Instead of scanning every vector, we navigate through the graph to reach the nearest vectors quickly.

Intuition Behind HNSW

Suppose we have embeddings:

e1
e2
e3
...
eN

Each embedding is a node in a graph.

Edges connect nodes that are similar vectors.

Example structure:

e1 ──── e5 ──── e9
 │       │
e2 ──── e6 ──── e10
 │       │
e3 ──── e7 ──── e11

When searching for nearest vectors, we walk through this graph toward nodes that are closer to the query.

Small World Graph Idea

HNSW is based on the concept of small world networks.

Small world networks have two properties:

  1. Local connections (connect to nearby nodes)
  2. Long range connections (jump across graph)

Example:

A ── B ── C ── D ── E
 \              /
   ──── Z ────

This allows:

  • fast traversal
  • logarithmic search time.

Hierarchical Structure

HNSW adds multiple layers of graphs. Upper layers contain fewer nodes and act as shortcuts.

Example structure:

Layer 2 (very sparse)
        A
       / \
      B   C

Layer 1 (medium density)
     A ── B ── C ── D

Layer 0 (dense)
  many nodes connected

Key idea:

  • Top layers allow large jumps
  • Bottom layers perform fine search

HNSW Index Construction

Each vector is inserted sequentially.

For each new vector:

  1. Start from the top layer entry point
  2. Navigate toward the nearest node
  3. Insert the vector into appropriate layers
  4. Connect it to nearest neighbors

Graph edges are limited to a fixed number:

M ≈ 16

Meaning each node connects to ~16 neighbors.

This keeps the graph sparse and efficient.

HNSW Search Algorithm

Given query embedding: $$q$$

Search works in stages.

Step 1: Start at top layer

Begin from a fixed entry node.

Navigate greedily toward nodes closer to the query.

Distance measure usually:

  • cosine similarity
  • L2 distance

Step 2: Move down layers

Once no closer node exists at the current layer:

move down to the next layer. Continue search.

Step 3: Final search

At the lowest layer (layer 0): perform local exploration to find top K nearest vectors.

HNSW Complexity

Search complexity approximately:

$$O(\log N)$$

Instead of brute force:

$$O(N)$$

This is why HNSW works well for millions or billions of vectors.

Parameters in HNSW

Important parameters:

M

Maximum number of edges per node.

Typical value:

M = 16

Higher values:

  • better recall
  • slower indexing.

efConstruction

Controls search breadth during index building.

Higher values improve index quality.

efSearch

Controls search depth during queries.

Higher value:

  • higher recall
  • slower search.

Hybrid Retrieval

Modern retrieval systems combine sparse and dense retrieval.

This is called hybrid retrieval.

Reason: Sparse retrieval captures:

  • exact tokens
  • numbers
  • names
  • code

Dense retrieval captures:

  • semantic similarity
  • paraphrases
  • natural language questions Combining both improves recall.

Hybrid Retrieval Architecture

Typical architecture:

Query
  ↓
Sparse Retriever (BM25)
  ↓
Dense Retriever (Embeddings)
  ↓
Combine Results
  ↓
Rerank
  ↓

Sparse Retrieval Component

Sparse retriever uses:

  • inverted index
  • BM25 scoring

Example systems:

  • Elasticsearch
  • Apache Lucene Output:
top K documents by BM25

Dense Retrieval Component

Dense retriever uses:

  • embeddings
  • ANN search (FAISS / HNSW)

Output:

top K nearest embeddings

Score Fusion

Results from both retrievers must be combined. Two common approaches exist.

Weighted Score Fusion

Scores are combined using:

$$Score = \alpha \cdot BM25 + (1-\alpha) \cdot DenseScore$$

Where:

  • \alpha controls sparse importance.

Example:

α = 0.5

Reciprocal Rank Fusion (RRF)

Another method combines ranks instead of scores.

Score is computed as:

$$RRF(d) = \sum_{m \in M} \frac{1}{k + rank_m(d)}$$

Where:

  • M is the set of retrieval systems
  • rank_m(d) is the rank of document d in system m
  • k is a constant (typically 60)

This method is robust and widely used in hybrid retrieval.

Practical Production RAG Systems

Real systems usually add another stage.

Retriever
   ↓
Reranker
   ↓
LLM

Rerankers are cross encoder models that evaluate query document pairs more precisely.

They improve retrieval quality significantly.

Adding a reranking step ensures the most contextually relevant chunks actually make it to the top of the LLM prompt, keeping the model focused. Combining hybrid search with a cross encoder makes for a solid, production ready RAG setup that scales pretty well.
