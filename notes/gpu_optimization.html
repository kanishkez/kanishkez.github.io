GPU Optimization

Training large models like Transformers takes a massive amount of GPU resources, and we often hit memory walls before compute walls. To get around this, we can use smart optimization tricks to trade compute for memory or speed up operations with reduced precision. This note covers three big techniques: gradient checkpointing, mixed precision training, and high precision accumulators.

Gradient Checkpointing

The Problem

In vanilla backpropagation, we store all intermediate activations during a forward pass, and then reuse them during the backward pass to compute gradients.

So, in a network of $L$ layers: $$x_0 \to f_1 \to x_1 \to f_2 \to x_2 \to \cdots \to f_L \to x_L.$$ The activations stored are $L$ as well.

During the backward pass, we load these stored activations and compute gradients.

Memory Cost

The memory cost for this is high, roughly $L$ times the activation size. As models get deeper, activation memory dominates, and you require a huge amount of memory to store these activations.

GPU Memory vs Computation

In GPUs, memory is scarce. Large activations eat VRAM. For example, a single $4096 \times 4096$ activation in FP16 is 32 MB. With 49 layers, that is about 1.5 GB for just one sample.

So, the bottleneck is memory and not compute. Modern GPUs deliver loads of TFLOPS. Forward computation is fast and cheap relative to memory usage. Thus, a good way to make GPU usage efficient is by using compute instead of storing everything in memory.

How Gradient Checkpointing Works

This is a practical solution to the GPU memory activation problem. The key idea is to store selected activations (checkpoints) only. During the backward pass, we can recompute the missing activations between checkpoints.

For $L$ layers, we might checkpoint at: $$0, k, 2k, 3k, \dots, L.$$ We store activations only at these checkpoints.

During the forward pass:

  1. Compute activations as usual.
  2. Store only checkpoint activations.
  3. Discard all others.

During the backward pass: For each segment, rerun the forward pass inside that segment to recompute the missing activations using the checkpoints.

With this approach, we only store $O(\sqrt{L})$ activations instead of $O(L)$. Memory reduces from linear to sublinear. Compute increases by 20 to 30%, but this is acceptable and much cheaper than running out of memory.

Mixed Precision Training

Mixed precision training means we train a model using multiple floating point precisions at the same time.

We use:

  1. FP16 or BF16 for most forward and backward passes and matmuls.
  2. FP32 to hold master weights.
  3. Loss scaling to prevent underflow.

What exactly happens

The master weights are stored in FP32. The GPU keeps a full precision copy of the weights for stability.

Forward Pass Before each forward pass, we copy the master weights into FP16 for computation. Activations, matmuls, and convolutions all happen in FP16.

Backward Pass Gradients are computed in FP16, but these gradients are not applied directly to the FP16 weights. Instead, they are used to update the FP32 master weights.

FP16 is not enough for everything because its dynamic range is limited, meaning small gradients might underflow to zero. For example, if a gradient is: $$1 \times 10^{-7}$$ FP16 cannot represent it as a normal number, and it might get flushed to 0.

Loss Scaling

To prevent tiny gradients from becoming zero, we scale up the loss. We multiply the loss by a scaling factor $S$: $$L_{\text{scaled}} = S \cdot L$$ Then, during backpropagation, the gradients are computed in scaled form. Before applying these gradients to the FP32 master weights, we undo the scaling: $$\nabla L = \frac{\nabla L_{\text{scaled}}}{S}$$ Loss scaling can be static, where we use a fixed constant. Dynamic loss scaling also exists, where the scale factor increases or decreases during training depending on gradient values to avoid underflow and overflow.

Accumulators

When GPUs perform a matrix multiplication: $$C = A \cdot B$$ They do not compute it one element at a time. Each element is a sum of many products: $$C_{ij} = \sum_{k=1}^{K} A_{ik} \cdot B_{kj}$$ The accumulator is the variable that stores this running sum. Accumulators are stored as FP32 instead of FP16 even if we are working with FP16 weights.

This is because even if we use FP16 weights, the sum of products needs FP32 precision to prevent loss of precision or underflow. For example, multiplying two numbers increases the required precision bits: $$2.02 \times 2.02 = 4.0804$$ So we use FP32 precision for accumulating the running sum. After we complete the matrix multiplication, we store the result back as FP16.

The accumulator is not for storing the final result, it is for computing the result correctly in high precision.

Similarly, optimizer states (like running mean, momentum, and the average of Adam) must remain in FP32 because they require high precision.

So that is the basic layout of how we keep GPU training efficient without sacrificing model convergence. By combining gradient checkpointing to save VRAM, mixed precision to accelerate matmuls, and high precision accumulators to keep things stable, we can scale up training significantly.
