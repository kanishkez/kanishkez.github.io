Mixture of Experts

Dense language models waste a massive amount of compute because they run every single token through every single parameter. Mixture of experts addresses this by using conditional computation to only activate a tiny fraction of the network for each token. This lets us scale the model's capacity and parameters without blowing up our training and inference costs.

Core Idea

Modern LLMs are powerful but computationally expensive since every token is processed by the entire dense network. Very often, when a token is processed, we don't need all of the activations or all of the weights.

Research has shown that for most tokens, only a small subset of weights (like 5%) is actually needed. The remaining 95% of compute is basically wasted.

To solve this, Mixture of Experts was introduced to process tokens using conditional computation.

Mixture of Experts are feed forward layers used as experts, where each token gets processed by an expert layer based on its domain instead of being processed by the entire dense model.

So, we have a bunch of experts that process tokens separately. When a token comes in, we decide which expert to send it to and process it.

$$y = \sum_i G(x)_i E_i(x)$$ The multiplication $G(x)_i E_i(x)$ means that the expert's opinion is only as valuable as the gate's trust in it.

Conditional Computation

The key idea is that we have a function $f(x,\theta)$, but we don't want to evaluate all parameters for every $x$.

So instead of one function, we define a family of functions: $$\{f_{1}, f_{2}, \dots, f_{E}\}$$ and we use a selection mechanism that decides which parameter is evaluated by which function.

$$z \sim p(z \mid x)$$

History

Early Mixture of Experts Original MoE (in 1991) used linear regressors as experts and used softmax as its gating network.

Theoretically MoE was right but practically training was unstable due to load imbalances and other problems.

2017 LSTMs were trained using a sparsely gated MoE layer.

Problems faced:

  1. Bad routing
  2. One expert dominates
  3. No parallelism and high overhead

GShard

GShard introduced:

  1. Top k routing
  2. Auxiliary load balancing loss: to prevent one expert from dominating, we add an auxiliary loss (acting as regularization)
  3. Expert parallelism

Switch Transformer

Here, we don't use Top k routing and use top 1 routing instead. One expert per token. No expert mixing.

Backprop: Only that expert updated.

Components

Experts

Usually feed forward networks with an activation function.

$$f_i(x) = W_{2,i}\,\sigma(W_{1,i}x)$$ Experts are the feed forward networks that tokens flow through inside a transformer layer. There are $E$ experts and usually processing is divided between experts. So generally, if we have $N$ tokens, the processing is divided between all experts individually.

Each token is only processed by one expert (unless you use top k routing or whatever).

Router / Gating Network

This produces the routing scores that decide which token goes to which expert.

Normally this is a simple linear layer where a token $x$ is multiplied by a matrix of weights. $$g(x) = xW_{g}$$ The router gives a score for every expert per token. So if you have 8 experts, we get 8 raw numbers. We then apply softmax to these scores and turn them into a probability distribution.

We then pick the routes using various routing algorithms.

There are many types of routing/gating.

Dense Gating

Here we apply softmax over all experts and compute all of them (very inefficient). This is only theoretical; nobody uses this practically.

Let's see what is actually used in modern MoEs.

Sparse Routing

Top k Routing We select the top k experts per token, picking them based on softmax probabilities.

$$g(x) = \text{TopK}(\text{softmax}(xW_g), k)$$ Issues:

  • Non differentiability
  • Load imbalance
  • Expert collapse

Switch Routing (Used in Switch Transformer) We use Top 1 routing here.

Each token gets assigned only one expert. It is simpler, faster, and more stable.

Router outputs scores and picks the argmax expert.

Routing

At the very beginning, the router's weights and experts' weights are all randomly initialized. The router picks experts randomly.

There is no specialization yet. The magic happens through backpropagation.

  • The router randomly sends a "math token" to an expert in the forward pass.
  • The gradient flows back through the model, and both the expert and the router learn.
  • Expert 3 becomes slightly better at math.
  • The router now sends math like problems to Expert 3 only.

That is how specialization learning happens to experts.

Expert Capacity

Each expert has a capacity: $$C = \left\lceil \frac{N \cdot k}{E} \cdot \text{capacity\_factor} \right\rceil$$ If too many tokens go to one expert, they are either rerouted or dropped.

$$\text{Expert Capacity} = \left( \frac{\text{tokens per batch}}{\text{number of experts}} \right) \times \text{capacity factor}$$

Load Balancing Loss

This is an auxiliary loss added to the original task loss which encourages uniform expert usage.

$$\mathcal{L}_{lb} = E \sum_{i=1}^E f_i p_i$$ Where:

  • $f_i$: fraction of tokens routed to expert $i$
  • $p_i$: average router probability for expert $i$

This is entropy maximization under constraints.

The ideal value for both $f_i$ and $p_i$ should be $\frac{1}{E}$ so that the router uses all experts equally and each expert receives the same amount of tokens as the others.

It is important to balance both so that routing is stable.

Working

  1. Token enters the transformer block (attention happens and we get the suitable output).
  2. Router computes scores and assigns an expert for the token. $$s = xW_g \quad\Rightarrow\quad i^* = \arg\max s$$
  3. We check the capacity of the expert. If it is full, we either drop the token, send it to a backup expert, or mask the gradient (depending on what we choose).
  4. Expert computation. Token is sent to the feed forward layer and is computed according to the expert. $$y = W_{2,i^*} \sigma(W_{1,i^*} x)$$
  5. Tokens are gathered back into the original order.

Backpropagation

Gradients only flow through the selected expert, the router parameters, and the load balancing loss. All others get zero gradient.

Parallelism

Note: Devices refer to training servers/GPUs etc.

Parallelism in large language models is the use of multiple processing units (like GPUs) to simultaneously execute different parts of the model or process data in parallel (during training and inference).

Types of Parallelism

Data Parallelism (DP)

What it is

  • Same model replicated on multiple devices
  • Each device processes different data
  • Gradients are averaged

The data is split into several shards, and every shard is allocated to each device, which equals parallel training in the dimension of the data batch. Every device has an intact copy/replica of the model, and the parameters of the model are updated based on the data split on each device.

Limitation: $$\text{Max model size} \le \text{single GPU memory}$$

Model Parallelism (MP)

Split the model itself across devices.

Tensor Parallelism

  • Split weight matrices (e.g. columns of FFN)
  • All GPUs compute every token

Pipeline Parallelism

  • Different layers on different devices
  • Microbatching required

Expert Parallelism (MoE's native parallelism)

Key idea

  • Experts are independent
  • Each expert lives on a different device
  • Tokens are routed to experts

Instead of splitting a matrix, we send different tokens to different networks present on different devices.

Hybrid Parallelism

Modern MoE LLMs use all of them:

  • Data parallelism across nodes
  • Tensor parallelism inside experts
  • Expert parallelism across experts
  • Pipeline parallelism across layers

Training parallelism maximizes throughput; inference parallelism minimizes latency under memory constraints.

Role of MoE in LLMs

It is important to revisit what the attention block does in a transformer.

For each token, it collects relevant context from the sequence and assigns probabilities/attention to other tokens in the sequence. So, for each token it tells how much it should focus on other tokens in the same sequence when generating an answer.

This is what an attention block does: it gathers context and relates one token to the other.

Feed forward networks look at this modified token (with attention and stuff, basically the hidden state after the attention block) and decide what token to generate next.

The actual generation of tokens happens inside the feed forward network.

The attention block doesn't understand or store facts; its job is to look at words and see how they relate to each other.

The FFN is where actual knowledge building happens. It stores facts, grammar rules, and reasoning.

Important: Researchers discovered that even in dense models, FFNs were already acting like experts. If you looked at a dense model, you would find that for a math question, only 10% of the neurons were doing the actual work.

So, MoE came to be by making these FFNs into specialized blocks and allowing these blocks to learn domain specialization.

In modern LLMs, MoEs hold most of the parameters instead of attention blocks because that is where most of the computation happens to see what token will be generated next.

Expert weights store facts, skills, and patterns, not attention blocks.

Everything that a model learns, like facts and logic, is stored in the expert blocks, not attention blocks.

Attention blocks learn:

  • Syntax
  • Alignment
  • Induction heads
  • Copy mechanisms
  • Positional logic

Attention blocks are only for contextualization.

MoE expert blocks are where computation and next token generation take place.

Ultimately, attention handles the relationships between tokens while mixture of experts does the heavy lifting of storing actual facts and generating the next token. Splitting FFNs into specialized experts lets us scale parameter counts to trillions without blowing up the compute budget.
